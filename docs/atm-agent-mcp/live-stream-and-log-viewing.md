# Live Stream and Log Viewing MVP

Status: Draft (implementation guidance)
Owner: `arch-ctm`
Scope: `atm-agent-mcp` + `atm-tui` UX integration for session watching and structured log viewing

## 1. Goals

1. Provide real-time Codex stream visibility in `atm-tui` with minimal new infrastructure.
2. Keep continuous high-rate stream traffic out of `atm-daemon`.
3. Preserve clear provenance for three input sources:
   - primary client prompt/steer (`arch-atm` style),
   - ATM mail injection (`arch-atm@atm-dev` style),
   - local user steer (`randlee` style).
4. Deliver practical structured log viewing using off-the-shelf tools.

## 2. Architecture Decision (MVP)

- `atm-agent-mcp` remains the stream owner.
- `atm-tui` watches a selected session by attaching directly to proxy stream output.
- `atm-daemon` receives lifecycle/state events only, not continuous stream deltas.
- This is an explicit exception to daemon fan-in for high-rate render transport only; it does not change daemon ownership of structured logs/state.

Rationale:
- lower implementation complexity,
- avoids daemon throughput/backpressure coupling,
- preserves existing daemon role as control/lifecycle hub.

## 3. Watch Stream Contract

### 3.1 Session watcher lifecycle

- `watch.attach(agent_id)`:
  - registers a watcher sink for one active session,
  - replays bounded buffer (see 3.2),
  - switches to live streaming.
- `watch.detach(agent_id)`:
  - removes watcher sink,
  - proxy resumes drop-on-no-watcher behavior for view-only deltas.

MVP constraint:
- one watcher sink per session.

### 3.2 Replay buffer

- Per-session ring buffer retained in proxy memory.
- Default size: `50` rendered lines/events.
- Attach behavior:
  1. emit replayed items oldest->newest,
  2. switch to live feed.

Notes:
- Buffer is for UX continuity while switching sessions.
- This does not replace durable history in rollout/log files.

### 3.3 Event subset for MVP rendering

Use a minimal, stable subset from Codex stream notifications:

- `turn/started`
- `item/agentMessage/delta`
- `item/reasoning/textDelta` (optional if reasoning panel enabled)
- `item/commandExecution/outputDelta`
- `item/completed`
- `turn/completed`
- `thread/status/changed`

Unknown events:
- ignore for rendering,
- count + log as debug telemetry.

## 4. Source Attribution Model

Every injected input and corresponding turn metadata should carry:

```json
{
  "source": {
    "kind": "client_prompt | atm_mail | user_steer",
    "actor": "arch-atm | arch-atm@atm-dev | randlee",
    "channel": "mcp_primary | mail_injector | tui_user"
  }
}
```

Rules:
- `client_prompt`: orchestrator-origin prompt/steer from primary MCP client.
- `atm_mail`: mail delivery turns generated by proxy.
- `user_steer`: local direct steering from `atm-tui` controls.

UI expectation:
- show concise origin marker on turn boundaries and in status rows.

## 5. Daemon Event Boundary

Send to daemon:
- session start/end,
- turn start/completion/idle transitions,
- stream error summaries,
- dropped-event counters (periodic).

Do not send to daemon:
- token/message delta streams,
- per-chunk command output deltas.

## 6. Structured Log Viewing MVP

### 6.1 Sink and schema

- Keep canonical JSONL logs (per requirements Section 4.6).
- Include correlation fields:
  - `team`, `agent`, `session_id`, `request_id`, `correlation_id`,
  - `source.kind`, `source.actor`,
  - `action`, `outcome`, `error`.

### 6.2 Viewer choice

Recommended MVP viewer: **Seq**.

Operational model:
1. keep JSONL as source-of-truth,
2. ship events to Seq (Vector/Fluent Bit/OTel collector),
3. define saved queries/dashboards for:
   - session timeline,
   - failures/timeouts,
   - mail-injection audit,
   - source-kind mix (`client_prompt` vs `atm_mail` vs `user_steer`).

## 7. Implementation Phasing

### Phase 1 (minimal)

- Add watcher attach/detach plumbing in `atm-agent-mcp`.
- Add per-session 50-line replay buffer.
- Add live watch pane in `atm-tui` using minimal event subset.
- Add source metadata tags for all three input paths.

### Phase 2 (hardening)

- Expand event coverage and rich formatting (code blocks, context usage display).
- Add optional multi-watcher fanout.
- Add stronger compatibility tests against Codex protocol drift.

## 8. Open Design Questions

1. Should replay buffer be line-based, event-based, or dual-mode?
2. Do we expose context-window usage in MVP if not uniformly available across transports?
3. Should `atm-tui` retain a local ephemeral buffer per viewed session in addition to proxy replay?
